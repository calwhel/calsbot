ISSUE 1 — BACKGROUND TASKS BLOCKING EVENT LOOP
═══════════════════════════════════════

PROBLEM: Any background task using time.sleep() instead of 
await asyncio.sleep() will completely freeze the FastAPI event 
loop, blocking ALL incoming requests while it sleeps.

CHECK AND FIX:
- Find every background task and async function in the codebase
- Replace ANY occurrence of time.sleep() inside async functions 
  with await asyncio.sleep()
- Replace ANY occurrence of time.sleep() inside asyncio tasks 
  with await asyncio.sleep()
- Check keepalive.py, background_tasks.py, main.py, and any 
  file touched in the recent performance upgrades
- Also check for any threading.Event().wait() or blocking I/O 
  inside async functions — these all block the event loop

The fix is simple but critical:
  WRONG:  time.sleep(240)
  RIGHT:  await asyncio.sleep(240)

ISSUE 2 — SYNCHRONOUS DB CALLS INSIDE ASYNC ROUTES
═══════════════════════════════════════

PROBLEM: SQLAlchemy by default is synchronous. If async FastAPI 
route handlers call synchronous SQLAlchemy session.execute() 
directly, they block the event loop for the entire duration of 
the DB query — freezing all other requests.

CHECK AND FIX:
- Audit every FastAPI route handler that is defined as async def
- For any route that calls synchronous SQLAlchemy operations, 
  wrap the DB call in run_in_executor to offload it to a thread:

  import asyncio
  from functools import partial

  # Instead of this (blocks event loop):
  result = session.execute(query)

  # Do this (runs in thread pool, non-blocking):
  loop = asyncio.get_event_loop()
  result = await loop.run_in_executor(
      None, 
      lambda: session.execute(query).fetchall()
  )

- Alternatively, if the route does not need to be async, 
  change it from "async def" to regular "def" — FastAPI will 
  automatically run it in a thread pool:

  # If you have no actual awaitable calls in the route,
  # this is simpler and correct:
  @app.get("/strategies")
  def get_strategies():  # NOT async def
      return fetch_strategies_from_db()

- Go through ALL routes in main.py and apply whichever 
  pattern is correct for each one

ISSUE 3 — CACHE REFRESH RUNNING TOO FREQUENTLY
═══════════════════════════════════════

PROBLEM: If the cache refresh background task has an error in 
its sleep interval, it may be running in a near-infinite tight 
loop — hammering the DB constantly and consuming all resources.

CHECK AND FIX:
- Find the refresh_heavy_caches() or equivalent background task
- Confirm the sleep interval is await asyncio.sleep(300) — 
  5 minutes between runs
- Add error handling so if the DB query fails, it still sleeps 
  before retrying — not retries in a tight loop:

  async def refresh_heavy_caches():
      while True:
          try:
              await asyncio.sleep(300)  # sleep FIRST
              data = await fetch_leaderboard()
              set_cache("leaderboard", data, ttl_seconds=360)
          except Exception as e:
              print(f"Cache refresh error: {e}")
              # still continues loop, next sleep will happen

- Note: sleep should be at the TOP of the loop, before the 
  DB query — so if startup is slow, it doesn't immediately 
  hammer the DB on launch

═══════════════════════════════════════

CHECK AND FIX:

  pool_size=5
  max_overflow=5
  pool_timeout=30
  pool_pre_ping=True
  pool_recycle=300
